# Development Guide

## Coding Style

Use:

use strict;
use warnings;

in every module.

Prefer small, focused methods over large ones.

Write code that explains itself before relying on comments.

---

## Error Handling

Use `warn` for user mistakes.

Examples:

* invalid arguments
* missing stack
* divide by zero

Reserve `die` for programmer errors or unrecoverable internal failures.

------------------------------------------------------------------------
Choosing Stack Operands vs Command Arguments
------------------------------------------------------------------------

When designing a new command, first determine whether it is part of the
programming language or part of the calculator environment.

Programming-language commands obtain all operands from the stack.

Examples

    +
    sqrt
    randint
    recall
    store

These commands must compose naturally inside executable values.

Environment commands interact with the calculator itself and may accept
command-line arguments.

Examples

    help
    tutorial
    loadfuncs
    savevars

This distinction allows executable values to remain simple while keeping
interactive commands convenient for REPL use.

As a general rule:

    If a command should work inside a Function, Token String or Code
    Block, it should obtain its operands from the stack.

---

## Commands

Every command should provide:

* help text
* category
* aliases (when appropriate)

Commands should be grouped into logical command modules.

---

## New Features

A new feature is not complete until it includes:

* implementation
* help text
* tutorial updates (if appropriate)
* automated tests
* roadmap update
* release notes

---

## Documentation

Tutorials should teach concepts before commands.

Examples should resemble actual calculator sessions whenever practical.

Favor examples using meaningful values over abstract numbers.

---

## Release Checklist

Before every release:

make syntax
prove -Ilib t
make test

Then:

* update tutorials
* update Roadmap.txt
* write release notes
* commit
* tag
* create backup

Never skip the test suite before a release.

---

## Git Tags

Each released version receives a tag.

Tags identify known-good releases.

Tags should not be rewritten after publication unless absolutely necessary.

If a tag must change:

* delete local tag
* delete remote tag
* recreate
* push

Published release tags should represent builds that pass the complete test suite.
